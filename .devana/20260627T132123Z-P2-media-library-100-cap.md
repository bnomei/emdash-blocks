# Media picker hard-caps library fetch at 100 items

## Finding

`loadMedia()` requests `/_emdash/api/media` with `limit=100` and no pagination or search. Assets beyond the first page never enter `items`, so they cannot be selected from the dropdown even when they exist in the library.

## Violated Invariant Or Contract

Any library asset the editor should attach must be reachable from the picker, or the UI must surface that results are truncated.

## Oracle

`url.searchParams.set("limit", "100")` is the only paging control. `selectMedia` resolves identities only against `items` and the current `selected` list.

## Counterexample

1. Media library contains 150 image items; the desired asset sorts after the first 100.
2. `loadMedia()` stores only the first 100 in `items`.
3. User opens the media select; asset #101+ is missing.
4. If that asset was previously stored, preview still works via `selected`, but after Remove the user cannot re-add it from the picker.

## Why It Might Matter

Sites with large media libraries silently lose picker access to most assets. The remove-then-reselect path fails while read preview still works, producing inconsistent behavior depending on operation order.

## Proof

Dataflow trace: API returns ≤100 items → `setItems` → dropdown built from `items` → `selectMedia(identity)` → `items.find` miss → `if (!media) return` with no user feedback.

## Counterevidence Checked

`allItems` merges `selected` for display of already-persisted values. This is a catalog coverage gap, not identity collision or storageKey traversal.

## Suggested Next Step

Add pagination, search, or raise limit with explicit truncation messaging; ensure `selectMedia` can resolve off-page assets by id.
